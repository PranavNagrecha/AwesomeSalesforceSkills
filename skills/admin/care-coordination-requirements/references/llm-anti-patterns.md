# LLM Anti-Patterns — Care Coordination Requirements

Common mistakes AI coding assistants make when generating or advising on Health Cloud care coordination.

## Anti-Pattern 1: Recommending Manual CareGap Creation

**What the LLM generates:** Instructions to create CareGap records manually via the UI, standard Flow DML, or data import to represent quality gaps identified by a care coordinator.

**Why it happens:** LLMs assume all standard objects in Salesforce can be created via standard record creation operations. The constraint that CareGap records are system-generated and cannot be manually created is a Health Cloud-specific platform behavior not inferrable from general Salesforce knowledge.

**Correct pattern:**
CareGap records (API v59.0+) must be generated by an external clinical rules engine and ingested via FHIR R4 API or direct API from a payer analytics system. Design the care gap workflow around receiving and actioning CareGap records, not creating them.

**Detection hint:** If the solution includes `new CareGap()` in Apex, a Flow "Create Records" step for CareGap, or a data import template for CareGap objects, it is incorrectly assuming manual creation is possible.

## Anti-Pattern 2: Confusing CareBarrier with CarePlanProblem

**What the LLM generates:** Recommendations to use CarePlanProblem for SDOH barriers like "food insecurity" or "transportation barrier," or to use CareBarrier for clinical diagnoses like "uncontrolled hypertension."

**Why it happens:** Both objects represent problems the care team needs to address. LLMs conflate them because both appear in care coordination documentation. The distinction between SDOH social barriers and clinical care plan problems is a domain-specific nuance not apparent from object names alone.

**Correct pattern:**
CareBarrier = SDOH social determinant obstacles (food, housing, transportation). Links to Account + Case. Resolved via Task closure. CarePlanProblem = clinical diagnoses and health problems within an ICM care plan. Part of the ProblemDefinition → GoalDefinition → ActionPlanTemplate hierarchy. Use each object only for its intended domain.

**Detection hint:** If the solution uses CarePlanProblem for transportation or food security barriers, or CareBarrier for clinical diagnoses, the objects are being misapplied.

## Anti-Pattern 3: Omitting ICM Permission Set Requirements

**What the LLM generates:** Care coordination configuration steps that assign standard Health Cloud user permission sets but omit the HealthCloudICM permission set required for ICM objects (ClinicalServiceRequest, CareGap, CareBarrier, CareEpisode).

**Why it happens:** Permission set requirements are a runtime access control detail not visible in the object schema. LLMs often focus on configuration steps and miss permission set assignment requirements, especially for Health Cloud's layered permission model.

**Correct pattern:**
All users who create, view, or update ICM objects (ClinicalServiceRequest, CareGap, CareBarrier, CareEpisode, and related) require the HealthCloudICM permission set. This applies to care coordinators, care team members, integration users, and any automated process users.

**Detection hint:** If the permission set assignment section covers only "Health Cloud" or "Health Cloud Platform" permission sets without mentioning HealthCloudICM, the ICM permission requirement is missing.

## Anti-Pattern 4: Assuming Care Coordination for Slack is Included in Health Cloud

**What the LLM generates:** Architecture designs that include Slack notifications for care team handoffs and referral alerts as a standard Health Cloud feature, without noting that this is a separately licensed add-on.

**Why it happens:** Salesforce markets Slack integration broadly and LLMs know Slack and Health Cloud are compatible products. The add-on licensing distinction is a commercial detail not inferrable from product documentation about capabilities.

**Correct pattern:**
Care Coordination for Slack is a separately licensed add-on, not included in base Health Cloud. The Slack + Health Cloud integration must also already be deployed before the add-on can be enabled. Confirm both the base integration and the add-on license are in place before designing Slack-dependent workflows.

**Detection hint:** If the design includes Slack-based care team alerts or notifications without noting the add-on licensing requirement, the license assumption is unverified.

## Anti-Pattern 5: Designing CareBarrier Without Related Case

**What the LLM generates:** CareBarrier creation workflows that link only to the patient Account record, without creating a related Case for care coordination tracking.

**Why it happens:** CareBarrier has an Account lookup that is intuitively the primary relationship. The Case relationship requirement for the full care coordination workflow — where Tasks for barrier resolution are linked to the Case — is a workflow design detail not apparent from the object definition alone.

**Correct pattern:**
CareBarrier records should be created with both an Account (patient) lookup and a related Case for coordination tracking. The Case enables Task assignment for intervention actions and provides the care coordinator with a unified coordination view. Create the Case first, then create the CareBarrier linked to both Account and Case.

**Detection hint:** If the CareBarrier creation workflow does not include Case creation or Case reference, the barrier resolution workflow will lack the Case-based Task management structure.
