# Gotchas — Care Coordination Requirements

Non-obvious Salesforce platform behaviors that cause real production problems in this domain.

## Gotcha 1: CareGap Records Cannot Be Created Manually

**What happens:** Any attempt to create a CareGap record via the UI, standard Flow DML, bulk import, or direct API insert from a non-system user will fail with an insufficient privileges error or simply produce no record.

**When it occurs:** When process designs include manual care gap entry by care coordinators, data migration of legacy quality measure gaps, or bulk import of initial care gap data. All of these fail because CareGap is a system-generated object designed to receive data from external clinical rules engines or FHIR R4 API ingestion.

**How to avoid:** At requirements design, explicitly document that care gaps will be generated by an external clinical rules engine and ingested via integration. Do not design any manual creation workflow for CareGap records. If legacy care gap data needs to be migrated, it must be done via the FHIR R4 API or a dedicated integration, not bulk data import tools.

## Gotcha 2: ICM Has Two Separate Setup Checkboxes

**What happens:** Enabling only one of the two ICM checkboxes causes ICM objects from the unenabled section to be unavailable. For example, enabling "Managing Care Plans" but not "Calculating Care Gaps" makes CareGap objects inaccessible.

**When it occurs:** During initial Health Cloud org setup, admins enabling ICM may see two checkboxes and enable only one, assuming they are sequential steps rather than independent feature gates.

**How to avoid:** Always enable both ICM checkboxes in Setup: (1) Enable Integrated Care Management for Managing Care Plans, and (2) Enable Integrated Care Management for Calculating Care Gaps. Verify both are enabled in Setup before testing ICM functionality.

## Gotcha 3: Care Coordination for Slack Is a Separate Licensed Add-On

**What happens:** Teams design Slack-based care team coordination workflows (e.g., care team alerts, referral notifications via Slack) and discover after design or even after implementation that Care Coordination for Slack is a separately purchased add-on that is not included in base Health Cloud licensing.

**When it occurs:** During requirements gathering when stakeholders assume that Health Cloud + Slack integration automatically enables care coordination workflows. The two are compatible but the Slack care coordination capabilities are licensed separately.

**How to avoid:** At project inception, confirm with the Salesforce account team whether the Care Coordination for Slack add-on is included in the org's licensing. Also confirm that the base Slack + Health Cloud integration (not the add-on) is already deployed. Design Slack workflows only after both prerequisites are confirmed.

## Gotcha 4: CareBarrier Links to Account AND Case — Both Are Required

**What happens:** CareBarrier records created with only an Account lookup (patient) but no Case reference fail to appear in the expected care coordination views and may trigger validation errors depending on org configuration.

**When it occurs:** When the SDOH barrier workflow design creates CareBarrier records at the point of social screening without also creating a related Case for care coordination tracking.

**How to avoid:** Design the SDOH barrier creation process to include Case creation: when a CareBarrier is identified, create a related Case to track the community resource intervention. Link both the Account (patient) and the Case to the CareBarrier record. This enables the full care coordination workflow including Task assignment and resolution tracking.
