# Gotchas — FHIR Data Mapping

Non-obvious Salesforce platform behaviors that cause real production problems in this domain.

## Gotcha 1: Clinical Objects Are Invisible Until the Org Preference Is Enabled — No Warning Given

**What happens:** If the FHIR-Aligned Clinical Data Model org preference (Setup > FHIR R4 Support Settings) is not enabled, objects like CareObservation, HealthCondition, CarePlanDetail, PersonName, ContactPointPhone, and ContactPointAddress simply do not appear in the schema. Queries return "entity type not found." There is no setup warning, no banner in Setup, and no validation error at configuration time — the org behaves as if the objects do not exist.

**When it occurs:** Any time a practitioner or middleware system attempts to query or write to FHIR-aligned clinical objects in an org where the preference has not been enabled. This catches people after org refreshes (sandbox refreshes often reset preferences), after new org provisioning, and in CI/CD pipelines where org state is assumed rather than verified.

**How to avoid:** Add a prerequisite validation step at the top of every FHIR migration runbook and every automated pipeline: query `FhirR4SupportSettings` metadata or perform a describe call on `HealthCondition` before any data operations. Fail fast with a clear message if the object is absent. Document in sandbox refresh procedures that this preference must be re-enabled after every refresh.

## Gotcha 2: condition.code Is Required in Health Cloud But Optional in the FHIR R4 Spec

**What happens:** FHIR R4 defines `Condition.code` as 0..1 — it is optional by specification. Salesforce Health Cloud's HealthCondition object enforces a required code reference. If middleware faithfully passes through code-less Condition resources, the DML insert fails. In batch loads via Bulk API, the error is reported at the job level and may be difficult to correlate back to the specific source resource.

**When it occurs:** This bites teams that are migrating from EHRs with loose clinical coding practices (e.g., problem lists where historical entries were created before coding mandates were in place) or from systems that store diagnosis descriptions as free text without a formal code. It also occurs when FHIR bundles are generated by older EHR adapters that do not enforce coding requirements.

**How to avoid:** Run a pre-load audit across all Condition resources in the source bundle. Separate resources into "coded" and "code-less" groups before any load operation. Load the coded group first. Route code-less resources to a clinical quarantine queue that requires a human coder to assign at minimum a SNOMED CT or ICD-10-CM code before the second load attempt. Never assign a placeholder or dummy code automatically — this creates incorrect clinical data.

## Gotcha 3: careTeam on CarePlan Is Not a Direct Object — It Resolves to Case Teams on the Parent Case

**What happens:** FHIR CarePlan includes a `careTeam` array referencing FHIR CareTeam resources. There is no careTeam field or CareTeam lookup on the Health Cloud CarePlan SObject. The clinical team is modeled through Salesforce Case Teams on the Case record that is the parent of the CarePlan. Practitioners who look for a careTeam field on CarePlan, or who try to create a custom junction object to replicate the FHIR structure, either fail to find the correct location or create a shadow data structure that is not recognized by Health Cloud features.

**When it occurs:** This affects any integration that includes FHIR CarePlan resources with populated careTeam references — common in care coordination and transitions-of-care use cases. It is especially non-obvious because the FHIR spec and Health Cloud documentation use the term "care team" but refer to structurally different implementations.

**How to avoid:** In middleware, resolve FHIR CareTeam participants to Salesforce User or Contact records. After loading the CarePlan, identify the parent Case record. Add the resolved participants as Case Team members using the `CaseTeamMember` SObject with the appropriate `TeamRoleId`. Note that practitioners without a Salesforce User account must be added as Contact-type team members — the Case Team supports both, but the `MemberId` field type differs. Validate the team membership by querying `CaseTeamMember` for the parent Case after load.

## Gotcha 4: PersonName, ContactPointPhone, and ContactPointAddress Are Child Records — Not Fields on the Person Account

**What happens:** FHIR Patient demographics (name, phone, address) do not map to fields on the Person Account SObject. They are stored in separate child SObjects: PersonName, ContactPointPhone, and ContactPointAddress. Each child record is linked to the Person Account via a ParentId lookup. A FHIR Patient with one name, two phone numbers, and one address requires four SObject records: one Person Account plus three child records. Practitioners who write patient demographics directly to `Account.FirstName`, `Account.LastName`, or `Account.Phone` create records that appear correct in the standard Salesforce UI but are invisible to Health Cloud clinical APIs and timeline views.

**When it occurs:** Most commonly during initial integration design when developers inspect the Person Account schema, find standard name and phone fields, and assume those are the correct targets. Also occurs after org upgrades when a pre-FHIR-model integration continues writing to Account fields without being updated.

**How to avoid:** Never write FHIR Patient demographic data to Account or Contact fields directly. Always create the child records (PersonName, ContactPointPhone, ContactPointAddress) after the Person Account upsert, using the Account Id as ParentId. Test by querying the Health Cloud Patient Summary API endpoint — if the demographics do not appear there, they are in the wrong records.
