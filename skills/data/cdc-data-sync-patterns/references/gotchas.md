# Gotchas — CDC Data Sync Patterns

Non-obvious Salesforce platform behaviors that cause real production problems in CDC replication pipelines.

## Gotcha 1: Gap Events Have No Field Data — Only recordIds

**What happens:** When Salesforce emits a `GAP_CREATE`, `GAP_UPDATE`, `GAP_DELETE`, `GAP_UNDELETE`, or `GAP_OVERFLOW` event, the event body contains no field values. The only meaningful content is the `ChangeEventHeader`, which includes `changeType`, `recordIds`, and `commitTimestamp`. Any code that reads field values from a gap event gets null or absent values.

**When it occurs:** Gap events are generated when: (1) a record change exceeds the 1 MB event size limit, (2) a bulk DML operation runs through a code path that bypasses the application server (common with Bulk API operations that use the asynchronous processing pipeline), or (3) an internal Salesforce error prevents full event generation. Gap events are not rare in orgs with active Bulk API usage.

**How to avoid:** Check `changeType` for the `GAP_` prefix before touching any field data. Route gap events to a REST API fallback path that queries current record state using `recordIds` from `ChangeEventHeader`. The REST response provides the authoritative current field values. Use `commitTimestamp` from the gap event header as a staleness guard when writing the queried values to the external datastore.

## Gotcha 2: Salesforce Has No Per-Subscriber Cursor — External replayId Persistence Is Mandatory

**What happens:** If a subscriber restarts without a stored `replayId`, it can only subscribe at tip (`-1`) or replay the full 72-hour window (`-2`). There is no way to ask Salesforce "where did this subscriber leave off?" — the platform simply does not track individual subscriber positions. Any records changed during the gap between the last processed event and the reconnection are permanently missed unless the `replayId` was persisted externally.

**When it occurs:** Any subscriber restart without external `replayId` persistence: process crash, container restart, deployment, scheduled maintenance. Even a brief outage of seconds can cause missed events in high-velocity orgs.

**How to avoid:** Persist the `replayId` to durable external storage (a database row, Redis key, or config file) after every successfully committed event batch — not just on shutdown. On reconnect, read the stored `replayId` and pass it to the subscription call. Treat the persistence write as part of the transaction: do not acknowledge events as processed until the `replayId` is durably stored.

## Gotcha 3: Offline More Than 72 Hours Makes the Stored replayId Useless

**What happens:** A subscriber that was offline for more than 72 hours provides its stored `replayId` to the Pub/Sub API on reconnect. The API returns an error indicating the replay ID is outside the retention window — the events it points to have been permanently purged. There is no gradual degradation; the entire stored position is invalid and all events from that point to the current tip are unrecoverable via replay.

**When it occurs:** Any subscriber downtime exceeding 72 hours: infrastructure outages, extended planned maintenance windows, disaster recovery scenarios. The 72-hour clock is wall-clock time, not event time.

**How to avoid:** Build explicit detection for the "replay ID expired" error from the Pub/Sub API. When detected, route to the Bulk API re-sync path: run a full Bulk API 2.0 query job to extract current record state for all affected objects, upsert those records into the external datastore, then subscribe at tip (`-1`) to resume CDC. This re-sync is the only way to establish a consistent baseline after the retention window has expired.

## Gotcha 4: transactionKey + sequenceNumber, Not Timestamp, for Intra-Transaction Ordering

**What happens:** Multiple CDC events generated by a single Salesforce transaction share the same `commitTimestamp`. Using timestamp alone to sort and apply events within the same transaction produces a non-deterministic order, which can result in incorrect field values being applied to the external datastore (e.g., an intermediate state overwriting the final state).

**When it occurs:** Any transaction that generates multiple CDC events: a trigger that updates related records, a process that modifies parent and child objects, or a batch class that processes multiple records in a single transaction. This is more common than it appears in orgs with complex automation.

**How to avoid:** Group events by `transactionKey` and apply them in ascending `sequenceNumber` order within each group. Use `commitTimestamp` only for ordering across different transactions (different `transactionKey` values). Use the combination of `transactionKey` + `sequenceNumber` as an idempotency key to safely handle event redelivery without double-applying changes.

## Gotcha 5: CDC Does Not Capture Pre-Existing Records — Day-0 Load Is Always Required

**What happens:** A new CDC subscription only receives events for changes that occur after the subscription is established. Records that existed before the subscription started are never published as CDC events. A subscriber that begins consuming CDC events without first loading existing records will have an incomplete view of the object's data.

**When it occurs:** Any new CDC replication pipeline for objects that already have records. This is the default case for any production org migration or new integration.

**How to avoid:** Always precede a new CDC subscription with a day-0 full load via Bulk API 2.0. Record the tip `replayId` before starting the Bulk API job, complete the full load, then subscribe at that recorded `replayId` to ensure no events published during the load window are missed. Only after the day-0 load is complete and confirmed should the CDC subscriber switch to normal incremental processing.
