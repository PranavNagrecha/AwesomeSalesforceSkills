# LLM Anti-Patterns — Patient Data Migration

Common mistakes AI coding assistants make when generating or advising on Patient Data Migration.
These patterns help the consuming agent self-check its own output.

## Anti-Pattern 1: Loading Clinical Objects Before Person Accounts (Wrong Insert Order)

**What the LLM generates:** A Bulk API 2.0 job plan that loads EhrPatientMedication, PatientHealthCondition, or other clinical objects in the same phase or before Person Accounts, often because the user asks for the "full migration script" and the LLM generates alphabetical or arbitrary object order.

**Why it happens:** LLMs see Health Cloud objects as a flat list. Without knowledge of the required AccountId dependency chain, they default to alphabetical order, object grouping by type, or the order the user mentioned the objects. The dependency graph (Person Accounts → Clinical → Care) is not obvious from object names alone.

**Correct pattern:**
```
Phase 1: Account (Person Account, Patient record type)  ← MUST be first
Phase 2: EhrPatientMedication, PatientHealthCondition, PatientImmunization,
         PatientMedicalProcedure  ← only after Phase 1 is JobComplete
Phase 3a: CarePlan  ← only after Phase 1 is JobComplete
Phase 3b: CarePlanProblem, CarePlanGoal  ← only after Phase 3a is JobComplete
Phase 3c: CarePlanTask  ← only after Phase 3b is JobComplete
```

**Detection hint:** Flag any migration plan that puts clinical or care objects in the same phase or an earlier phase than `Account`. Look for missing phase-gate conditions (no polling loop between phases).

## Anti-Pattern 2: Assuming Platform-Generated Engagement History Can Be Bulk-Imported

**What the LLM generates:** A migration scope that includes "Care Plan Engagement History" or "Patient Outreach History" as importable objects, with suggested Bulk API 2.0 job configurations or Data Loader mappings for these records.

**Why it happens:** LLMs conflate "engagement history" as a concept (historical data) with "data records" (importable). Health Cloud's engagement engine generates engagement history as a platform side effect, not as a user-writable record type. The LLM has no signal that these are system-generated and lacks knowledge of the DML surface restriction.

**Correct pattern:**
```
# Exclude from migration scope:
# - Care Plan Engagement History (system-generated by engagement engine)
# - Field Audit Trail entries (system-generated, immutable)
# - Process/Flow execution logs

# Document the gap in the migration runbook:
# "Engagement history prior to [go-live date] is not available in Health Cloud.
#  Historical engagement data is retained in source EMR for audit reference."
```

**Detection hint:** Flag any migration plan that includes a job for `CareObservation`, engagement log objects, or audit trail objects as importable load targets. These require investigation before treating as in-scope.

## Anti-Pattern 3: Skipping Shield Platform Encryption Setup Before Data Load

**What the LLM generates:** A migration runbook that notes Shield Platform Encryption as a "post-migration security hardening step" or places it in a parallel track with the data load, rather than as a hard pre-requisite gate.

**Why it happens:** LLMs treat security configuration and data migration as independent workstreams. They often suggest running them in parallel to save time, not recognizing that HIPAA compliance requires encryption to be active before PHI arrives in the org, and that enabling encryption after the fact leaves unencrypted PHI in audit logs and feed history.

**Correct pattern:**
```
Pre-flight gate (all must pass before any data load):
1. BAA signed with Salesforce — confirm in contract records
2. Shield Platform Encryption active on PHI fields:
   - Account: PersonBirthdate, SSN__c, MRN__c (and any custom PHI fields)
   - EhrPatientMedication: MedicationName__c, Dosage__c
   - PatientHealthCondition: ConditionCode__c, DiagnosisDate__c
3. TLS 1.2+ enforced on all Bulk API 2.0 pipeline connections
4. Salesforce Data Mask applied to target sandbox
→ Only proceed to Phase 1 (Person Account load) after all four pass.
```

**Detection hint:** Flag any runbook or timeline that places "enable Shield Encryption" after or concurrent with any data load step. Shield must precede the first data load unconditionally.

## Anti-Pattern 4: Using Salesforce Record IDs Instead of External IDs for Cross-Object References

**What the LLM generates:** A migration pipeline that queries Salesforce for record IDs after Phase 1 (e.g., queries Account IDs for all loaded Person Accounts), builds an ID mapping table, and then uses those Salesforce IDs directly in Phase 2 clinical object CSVs.

**Why it happens:** LLMs are familiar with database foreign-key patterns. When translating a relational migration approach to Salesforce, they default to: load parent → get IDs → inject IDs into child records. They are less likely to know about Bulk API 2.0's native relationship-by-external-ID capability.

**Correct pattern:**
```csv
# WRONG — hard-coded Salesforce IDs are environment-specific and break
# on sandbox refresh or if records are re-inserted:
AccountId,Name,MedicationName__c
0014x000000XXXXX,Metformin 500mg,Metformin

# CORRECT — relationship-by-external-ID resolves AccountId from EMR_Patient_ID__c:
Account.EMR_Patient_ID__c,Name,MedicationName__c
MRN-001234,Metformin 500mg,Metformin
```

**Detection hint:** Flag any migration script or CSV template that contains 15- or 18-character Salesforce IDs as direct lookup values for clinical or care objects. These should be replaced with relationship-by-external-ID column headers.

## Anti-Pattern 5: Loading PHI Into Sandboxes Without Data Masking

**What the LLM generates:** A suggestion to use a "representative sample of real patient data" in the development or QA sandbox to validate field mappings, test the Bulk API 2.0 pipeline, or reproduce production data issues.

**Why it happens:** Using real data for integration testing is a standard pattern in non-HIPAA contexts. LLMs trained on general software engineering content apply this pattern without recognizing that HIPAA applies regardless of environment, and that sandboxes are frequently outside the scope of the HIPAA BAA.

**Correct pattern:**
```
For all pre-production migration testing:
- Use synthetic patient data: structurally valid, schema-matching, but with no
  real patient identifiers (generated names, DOBs, MRNs that do not map to
  real individuals).
- Apply Salesforce Data Mask to every sandbox org before the project starts.
- Confirm sandbox is covered by BAA if any real PHI is ever loaded.
- Reserve real PHI loads for the production migration window only.

Synthetic data generation example (Python stdlib):
  import uuid, random
  mrn = f"SYNTH-{uuid.uuid4().hex[:8].upper()}"
  dob = f"{random.randint(1940,2005)}-{random.randint(1,12):02d}-{random.randint(1,28):02d}"
```

**Detection hint:** Flag any suggestion to use "real data," "production data," or "live patient records" in a sandbox, development, or testing environment without explicit confirmation that Salesforce Data Mask has been applied and the sandbox is BAA-covered.

## Anti-Pattern 6: Assuming CarePlanTask Directly References CarePlan

**What the LLM generates:** A CarePlanTask CSV or insert script where the parent relationship field is mapped to a `CarePlan` ID, skipping the intermediate `CarePlanGoal` relationship.

**Why it happens:** The care plan hierarchy (Plan → Problem/Goal → Task) is intuitive as Plan → Task at first glance. LLMs generating Salesforce schema advice without full Health Cloud object reference knowledge assume a direct parent-child relationship from Task to Plan, mirroring generic project management data models.

**Correct pattern:**
```
Correct hierarchy:
  CarePlan
    └── CarePlanProblem (references CarePlan)
    └── CarePlanGoal    (references CarePlan)
          └── CarePlanTask  (references CarePlanGoal — NOT CarePlan)

CarePlanTask CSV:
  Goal_Ext_ID__c (as CarePlanGoal.Goal_Ext_ID__c),Subject,Status
  GOAL-001,Schedule follow-up appointment,Open
  GOAL-001,Review lab results with patient,Completed
```

**Detection hint:** Flag any CarePlanTask insert script or CSV that includes a column mapping to a CarePlan ID (e.g., `CarePlan__c`, `CarePlanId`, or `CarePlan.CarePlan_Ext_ID__c`) as the parent reference. The correct parent field resolves through CarePlanGoal.
