# Gotchas — Clinical Decision Support

Non-obvious Salesforce platform behaviors that cause real production problems in this domain.

## Gotcha 1: No Native Clinical Rules Evaluation Engine in Salesforce

**What happens:** ClinicalAlert and CareGap records do not appear in the care coordinator console after Health Cloud is configured and clinical data is loaded. Investigation reveals no records exist in either object.

**When it occurs:** When implementation teams design clinical decision support workflows assuming Salesforce will auto-generate alerts and gaps from clinical data, similar to how clinical rules engines work in purpose-built clinical systems (Epic, Cerner, Innovaccer).

**How to avoid:** Always define the explicit record creation mechanism for every ClinicalAlert type at design time. ClinicalAlert records require one of: (a) Apex trigger on a clinical data change event, (b) Record-Triggered Flow with DML on ClinicalAlert, or (c) FHIR R4 API ingest from an external system. For CareGap: external rules engine integration is required.

---

## Gotcha 2: CareGap Records Cannot Be Created via Standard DML

**What happens:** Apex code or Flow automation that attempts to insert CareGap records via standard `insert` DML or Flow "Create Records" action fails with an error indicating the operation is not permitted on this object.

**When it occurs:** When care coordinators or admins attempt to create CareGap records manually, or when Apex triggers attempt to auto-generate quality gaps without going through the correct FHIR API or system-generated mechanism.

**How to avoid:** Design all CareGap workflows around ingestion from external systems (payer quality analytics, clinical rules engines). CareGap records must arrive via FHIR R4 API or be generated by the system. Do not include CareGap creation in any manual workflow or Apex trigger.

---

## Gotcha 3: Apex Triggers on Clinical Objects Must Be Bulkified

**What happens:** During bulk clinical data operations (EHR nightly sync loading thousands of CareObservation records), the Apex trigger that creates ClinicalAlert records hits CPU time limits or DML limits and fails, causing clinical alert data loss.

**When it occurs:** When ClinicalAlert creation triggers are written with per-record patterns (not bulk-aware) that perform SOQL inside trigger loops or create individual DML statements per record.

**How to avoid:** All Apex triggers on clinical objects (CareObservation, HealthCondition, PatientMedication) must follow Salesforce trigger bulkification best practices: collect all records in `Trigger.new` first, perform bulk logic evaluation outside DML loops, use List-based DML for all inserts. Test with bulk data volumes (200+ records per trigger invocation) in sandbox.

---

## Gotcha 4: Business Rules Engine Is a Separately Licensed Add-On

**What happens:** The implementation team designs clinical protocol evaluation using the Business Rules Engine, then discovers it is not included in the Health Cloud license and requires a separate purchase.

**When it occurs:** During implementation when the team plans to use BRE for declarative rule-based ClinicalAlert creation (e.g., "if patient meets these clinical criteria, create this alert type") without confirming the BRE license.

**How to avoid:** At project inception, confirm whether the Business Rules Engine add-on is included in the license or needs to be purchased. If not licensed, implement equivalent logic in Apex. Document the BRE license requirement in the technical prerequisites checklist.
