# LLM Anti-Patterns — Clinical Decision Support

Common mistakes AI coding assistants make when generating or advising on clinical decision support in Health Cloud.

## Anti-Pattern 1: Assuming Health Cloud Has a Built-In Clinical Rules Engine

**What the LLM generates:** Architecture designs or implementation plans that describe Health Cloud automatically creating ClinicalAlert or CareGap records based on patient data changes, without any explicit creation mechanism.

**Why it happens:** Purpose-built clinical systems (Epic, Cerner, Innovaccer) have clinical rules engines that auto-evaluate patient data and generate alerts. LLMs apply this behavior pattern to Salesforce Health Cloud, which does not include a native clinical rules evaluation engine.

**Correct pattern:**
ClinicalAlert and CareGap are passive storage objects. No alerts or gaps auto-generate from clinical data. Alert creation requires explicit mechanisms: Apex trigger on clinical data change, Record-Triggered Flow with ClinicalAlert DML, FHIR R4 API ingest from external systems, or Business Rules Engine (if licensed).

**Detection hint:** If the CDS design shows clinical alerts appearing "automatically" or "when conditions are met" without an explicit Apex trigger, Flow, or integration step, the missing creation mechanism is a design gap.

## Anti-Pattern 2: Creating CareGap Records via Standard DML or Flow

**What the LLM generates:** Apex code or Flow automation that creates CareGap records when care coordinators identify quality gaps during patient outreach, treating CareGap like a standard object that supports standard DML.

**Why it happens:** CareGap looks like a standard insertable SObject from its schema. LLMs assume it can be created via standard `insert` DML without knowing the system-generated constraint.

**Correct pattern:**
CareGap records (API v59.0+) must be generated by external clinical rules engines and ingested via FHIR R4 API or system-generated processes. Standard DML inserts on CareGap from Apex or Flow are not the correct pattern. Design gap tracking workflows around CareGap ingestion from authoritative external systems.

**Detection hint:** If the solution includes `new CareGap()` with standard Apex DML or a Flow "Create Records" action targeting CareGap for manually identified gaps, the wrong creation pattern is being applied.

## Anti-Pattern 3: Non-Bulkified Apex Triggers on Clinical Objects

**What the LLM generates:** Apex trigger code on CareObservation or HealthCondition that creates ClinicalAlert records with SOQL inside the trigger loop, or that performs individual DML inserts per record rather than bulk inserts.

**Why it happens:** Simple Apex examples often show record-by-record processing for clarity. LLMs generate readable but non-bulkified trigger code without anticipating bulk clinical data loads from EHR integrations.

**Correct pattern:**
```apex
// Correct: Bulkified pattern
List<ClinicalAlert> alertsToCreate = new List<ClinicalAlert>();
for (CareObservation obs : trigger.new) {
    if (meetsAlertCriteria(obs)) {
        alertsToCreate.add(buildAlert(obs));
    }
}
if (!alertsToCreate.isEmpty()) {
    insert alertsToCreate;
}
```
Never perform SOQL queries or DML inside trigger loops. Collect all records first, evaluate in bulk, insert in bulk.

**Detection hint:** If the Apex trigger contains `insert new ClinicalAlert(...)` or SOQL inside a for-loop over `Trigger.new`, the trigger is not bulkified.

## Anti-Pattern 4: Designing CDS Alerts Without MuleSoft for EHR Integration

**What the LLM generates:** CDS Hooks integration designs where Salesforce's FHIR Healthcare API is registered directly as the CDS Hook service endpoint in the EHR.

**Why it happens:** Salesforce supports FHIR R4 and CDS Hooks is a FHIR-related standard. LLMs infer that Salesforce can natively serve CDS Hook responses.

**Correct pattern:**
There is no native Salesforce CDS Hook service endpoint. MuleSoft must act as the CDS Hook service: receive the POST from the EHR, query Salesforce ClinicalAlert/CareGap records, assemble the CDS card response, and return it to the EHR. Salesforce is the data store, not the CDS service.

**Detection hint:** If the CDS Hooks architecture shows Salesforce's FHIR Healthcare API URL as the CDS service endpoint without a MuleSoft layer, the architecture is incorrect.

## Anti-Pattern 5: Omitting Business Rules Engine License Check

**What the LLM generates:** Clinical decision support designs that recommend using the Business Rules Engine (BRE) for protocol evaluation without noting that BRE is a separately licensed add-on.

**Why it happens:** BRE is a Salesforce platform feature available in some editions and add-on packages. LLMs recommend it based on functional fit without knowing the licensing distinction.

**Correct pattern:**
Before designing any BRE-based clinical protocol evaluation, confirm the Business Rules Engine add-on is included in the org's Health Cloud license. If not licensed, implement equivalent protocol logic in Apex triggers or Decision elements in Flow. Document the BRE license dependency in the project's license prerequisites checklist.

**Detection hint:** If the CDS implementation plan includes BRE-based rule definitions without a licensing confirmation step, the license assumption is unverified.
