# Examples — Clinical Decision Support

## Example 1: Creating ClinicalAlert from Lab Result Threshold Breach

**Context:** A health system wants to automatically create a care coordinator alert when a patient's CareObservation record shows an A1c value above 9.0 (indicating poorly controlled diabetes).

**Problem:** The team assumed Health Cloud would automatically evaluate clinical data and create alerts. After implementing the FHIR integration that populates CareObservation records, no ClinicalAlert records appear — because there is no built-in rules engine.

**Solution:**
1. Create an Apex trigger on CareObservation with `after insert, after update`.
2. Implement the threshold evaluation in the trigger handler:
   ```apex
   List<ClinicalAlert> alertsToInsert = new List<ClinicalAlert>();
   for (CareObservation obs : trigger.new) {
       if (obs.ObservationCode == 'LOINC:4548-4' && // HbA1c LOINC code
           obs.ValueQuantity != null && 
           obs.ValueQuantity > 9.0) {
           alertsToInsert.add(new ClinicalAlert(
               PatientId = obs.PatientId,
               AlertType = 'LabResult',
               AlertSeverity = 'High',
               AlertMessage = 'A1c result ' + obs.ValueQuantity + '% exceeds 9.0 — clinical review required.',
               Status = 'Active'
           ));
       }
   }
   if (!alertsToInsert.isEmpty()) {
       insert alertsToInsert;
   }
   ```
3. Configure the care coordinator console to display ClinicalAlert records for the patient.
4. Define the alert acknowledgment workflow (care coordinator updates Status to Acknowledged).

**Why it works:** ClinicalAlert is the correct target object for protocol-based alerts in Health Cloud. The Apex trigger implements the clinical logic that Salesforce does not provide natively. Bulkified trigger pattern handles bulk CareObservation imports without hitting limits.

## Example 2: Designing CareGap Ingestion from a Payer Quality System

**Context:** A managed care organization has a payer quality analytics platform that identifies quality measure gaps (overdue mammograms, missing colorectal screenings, uncontrolled A1c). These need to appear in Salesforce Health Cloud for care coordinator outreach.

**Problem:** The care coordination team asked whether care coordinators can manually create CareGap records in the Salesforce UI for patients they identify in their outreach lists. This is not possible — CareGap records are system-generated and cannot be created via manual UI entry.

**Solution:**
1. Redesign the requirements: care gaps will be generated by the payer quality system, not manually created.
2. Configure the payer quality system to output quality gap data in FHIR R4 CareGap resource format.
3. Deploy a MuleSoft integration that runs nightly:
   - Queries payer quality system API for all open care gaps.
   - Maps each gap to a Salesforce CareGap record (patientId via Account ID lookup, measure code, gap status, gap period).
   - Calls Salesforce FHIR Healthcare API to create or update CareGap records.
4. Care coordinators see open CareGap records on patient pages.
5. When a care coordinator contacts a patient about a gap, they create a related Task linked to the CareGap record.
6. When the quality system confirms the gap is closed (e.g., patient received mammogram), the nightly MuleSoft job updates the CareGap status to Closed.

**Why it works:** The ingestion-driven approach is architecturally correct for CareGap records. The nightly integration keeps Salesforce CareGap records synchronized with the authoritative quality analytics system. Care coordinators act on gaps but do not create them.

## Anti-Pattern: Assuming Health Cloud Has a Built-In Clinical Rules Engine

**What practitioners do:** Design care coordinator workflows that assume Salesforce automatically creates ClinicalAlert records based on patient data (e.g., "Health Cloud should flag patients with no care plan as an alert automatically").

**What goes wrong:** ClinicalAlert and CareGap are passive data-model objects with no self-activating rules logic. No alerts appear in the care coordinator console unless Apex triggers, Flows, or external system integrations explicitly create the records. The care coordinator console shows an empty alert list for all patients.

**Correct approach:** Define the alert creation mechanism explicitly for each alert type: which clinical event or data condition triggers the alert, which Apex/Flow/integration creates the ClinicalAlert record, and what the alert fields should contain. ClinicalAlert records require an explicit creation action — they do not auto-generate from clinical data.
