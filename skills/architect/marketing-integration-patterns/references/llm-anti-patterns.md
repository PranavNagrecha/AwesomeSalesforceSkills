# LLM Anti-Patterns — Marketing Integration Patterns

Common mistakes AI coding assistants make when generating or advising on Marketing Cloud integration patterns. These patterns help the consuming agent self-check its own output.

## Anti-Pattern 1: Confusing Triggered Sends with Journey Injection

**What the LLM generates:** Code that calls `/interaction/v1/events` (the Journey Injection endpoint) for transactional single-message sends like order confirmations or password resets, or conversely, code that calls the Triggered Send endpoint for multi-step journey enrollment.

**Why it happens:** LLMs conflate the two patterns because both involve "sending a message via API." Training data contains both patterns without clearly distinguishing the architectural use cases. The LLM may default to whichever endpoint it has seen more frequently in code samples.

**Correct pattern:**

```
Triggered Send Definition endpoint — for transactional single-message sends:
POST /messaging/v1/messageDefinitionSends/key:<ExternalKey>/send

Journey Injection endpoint — for multi-step Journey enrollment:
POST /interaction/v1/events
```

Use Triggered Send when: one subscriber, one message, fire-and-forget, transactional (order confirmation, password reset).
Use Journey Injection when: the contact should enter a branching, multi-step sequence driven by behavior or time delays.

**Detection hint:** If the generated code uses `/interaction/v1/events` for a single transactional message, flag it. If it uses `/messaging/v1/messageDefinitionSends/` for Journey enrollment, flag it.

## Anti-Pattern 2: Missing or Incorrect `eventDefinitionKey` — Using Journey ID Instead

**What the LLM generates:** Journey Injection code that populates `EventDefinitionKey` with the Journey ID (a UUID from the Journey Builder URL) or the Journey name string, rather than the `eventDefinitionKey` generated by the REST API Entry Source.

**Why it happens:** LLMs see "Journey" and "ID" together in documentation and training examples and assume the Journey ID is the correct identifier. The `eventDefinitionKey` is a separate, less prominently documented identifier that is only visible in the Entry Source properties panel.

**Correct pattern:**

```json
// WRONG — do not use Journey ID or Journey name
{
  "ContactKey": "sub-12345",
  "EventDefinitionKey": "1a2b3c4d-5e6f-7890-abcd-ef1234567890",  // This is a Journey ID
  "Data": {}
}

// CORRECT — use the eventDefinitionKey from the REST API Entry Source
{
  "ContactKey": "sub-12345",
  "EventDefinitionKey": "APIEvent-abc123-def456-7890-abcdef123456",
  "Data": {}
}
```

The `eventDefinitionKey` always starts with `APIEvent-` for REST API Entry Sources. Journey IDs do not follow this format.

**Detection hint:** If `EventDefinitionKey` in generated code does not start with `APIEvent-`, or if the code fetches a "Journey ID" and uses it as the event key, flag it.

## Anti-Pattern 3: Missing OAuth 2.0 Client Credentials Setup — Assuming Username/Password or Legacy Auth

**What the LLM generates:** Marketing Cloud API integration code that authenticates using a username, password, and clientSecret (SOAP-style auth), or that uses an API key directly without a token acquisition step, or that re-authenticates on every single API call.

**Why it happens:** LLMs trained on older Marketing Cloud documentation (pre-2019) may default to legacy auth patterns. They may also conflate Marketing Cloud SOAP API auth (which used username/password/clientId in the SOAP header) with REST API auth (which requires OAuth 2.0 client credentials via the `/v2/token` endpoint).

**Correct pattern:**

```python
# WRONG — SOAP-style credentials do not work for REST
headers = {
    'Username': 'user@example.com',
    'Password': 'secretpassword',
    'clientId': 'abc123'
}

# CORRECT — OAuth 2.0 client credentials from Installed Package
token_resp = requests.post(
    f'https://<subdomain>.auth.marketingcloudapis.com/v2/token',
    json={
        'grant_type': 'client_credentials',
        'client_id': os.environ['MC_CLIENT_ID'],
        'client_secret': os.environ['MC_CLIENT_SECRET']
    }
)
token = token_resp.json()['access_token']
rest_url = token_resp.json()['rest_instance_url']
# Cache token for 18-19 minutes before refreshing
```

**Detection hint:** Flag any generated code that includes `Username`, `Password`, or raw API key in REST API headers. Flag any generated code that calls the token endpoint inside a per-request loop without caching.

## Anti-Pattern 4: Exceeding the 100-Contact Async Batch Limit Without Chunking

**What the LLM generates:** Code that passes a full list of contacts (potentially thousands) to the `/interaction/v1/events/async` endpoint in a single request, without splitting into chunks of ≤ 100.

**Why it happens:** LLMs often generate "batch" code that passes the entire input list to a batch endpoint, assuming the API handles internal pagination or truncation gracefully. The Marketing Cloud async event endpoint does not — it returns HTTP 400 for any payload exceeding 100 contacts.

**Correct pattern:**

```python
# WRONG — passes entire list, fails if len(contacts) > 100
requests.post(f"{rest_url}/interaction/v1/events/async", json={
    "EventDefinitionKey": event_key,
    "contacts": contacts  # Could be thousands of entries — will 400
})

# CORRECT — chunk into batches of 100
BATCH_SIZE = 100
for i in range(0, len(contacts), BATCH_SIZE):
    batch = contacts[i:i + BATCH_SIZE]
    requests.post(f"{rest_url}/interaction/v1/events/async", json={
        "EventDefinitionKey": event_key,
        "contacts": batch
    })
```

**Detection hint:** If generated code passes a variable-length list to `/interaction/v1/events/async` without a slicing or chunking step, flag it.

## Anti-Pattern 5: Using a Hardcoded Generic REST Instance URL Instead of the Tenant-Specific URL

**What the LLM generates:** API call code that uses a hardcoded base URL like `https://www.exacttargetapis.com` or `https://mcapi.salesforce.com` instead of the tenant-specific `rest_instance_url` returned in the token response.

**Why it happens:** LLMs see domain examples in documentation and generate calls against those example domains. The correct `rest_instance_url` is dynamic — it is returned as part of the OAuth 2.0 token response and is unique to each Marketing Cloud tenant (e.g., `https://mc<tenant-id>.rest.marketingcloudapis.com`).

**Correct pattern:**

```python
# WRONG — hardcoded generic domain
response = requests.post(
    "https://www.exacttargetapis.com/messaging/v1/messageDefinitionSends/key:mykey/send",
    ...
)

# CORRECT — use rest_instance_url from the token response
token_resp = requests.post(token_endpoint, json=credentials).json()
rest_url = token_resp['rest_instance_url']  # Tenant-specific URL
response = requests.post(
    f"{rest_url}/messaging/v1/messageDefinitionSends/key:mykey/send",
    ...
)
```

**Detection hint:** Flag any generated code that constructs Marketing Cloud REST API URLs from hardcoded domain strings rather than from the `rest_instance_url` field of the token response.

## Anti-Pattern 6: Recommending Custom API Integration When MC Connect Is the Right Tool

**What the LLM generates:** Apex callout code or external API integration design for syncing Salesforce CRM objects (Contact, Lead, Account, Campaign) into Marketing Cloud Data Extensions, when the org already has Marketing Cloud Connect installed.

**Why it happens:** LLMs default to API integration patterns when the prompt mentions "sync data between Salesforce and Marketing Cloud." Marketing Cloud Connect (MC Connect) provides native Synchronized Data Extensions (SDEs) that replicate CRM objects without custom code. LLMs underweight this option because it is a configuration-based solution that appears less often in code-centric training data.

**Correct pattern:**

```
WRONG: Custom Apex callout to Marketing Cloud REST API for each Contact update

CORRECT: Configure MC Connect Synchronized Data Extensions in Marketing Cloud Setup.
- Navigate to: Marketing Cloud > Administration > Salesforce Integration > Contact Builder
- Configure Synchronized Data Extensions for Contact, Lead, Account, CampaignMember
- SDEs refresh on schedule (15-minute default) or near-real-time for configured objects
- No custom Apex or API code required
```

**Detection hint:** If the generated solution involves Apex callouts or REST API calls from Salesforce core to Marketing Cloud purely for CRM object data sync (Contacts, Leads, Accounts), and the org has MC Connect, flag the recommendation and suggest SDEs instead.
