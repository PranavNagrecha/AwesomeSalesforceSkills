# Gotchas — Idempotent Integration Patterns

Non-obvious Salesforce platform behaviors that cause real production problems in this domain.

## Gotcha 1: "Publish Immediately" Fires Before the Transaction Commits

**What happens:** Platform Events with "Publish Behavior: Publish Immediately" (the default) are enqueued on the event bus at the moment `EventBus.publish()` is called in Apex, before the surrounding transaction commits. If the publishing transaction subsequently encounters a DML exception or is explicitly rolled back via a `Savepoint`, the event has already been dispatched. Subscribers receive the event, process it, and attempt to create or update records that reference a parent that was never persisted. This breaks causal consistency: the event is real but the data it describes is not.

**When it occurs:** Any time `EventBus.publish()` is called inside a transaction that also contains DML that could fail — for example, publishing a `Payment_Confirmed__e` inside the same transaction that inserts the `Payment__c` record. If the `Payment__c` insert fails a validation rule and the transaction rolls back, the `Payment_Confirmed__e` event has already been delivered to subscribers.

**How to avoid:** Change the Platform Event's "Publish Behavior" to "Publish After Commit" in Setup → Platform Events → [Event Name] → Edit. With this setting, the event is only enqueued if the surrounding transaction commits successfully. If the transaction rolls back, no event is published. This is the correct setting for any event that describes the outcome of a DML operation.

## Gotcha 2: Idempotency Key Regenerated on Each Retry

**What happens:** Code that generates a UUID or hash inside the callout execution path (inside `execute()` in a Queueable, or inside the invocable action body) produces a brand-new idempotency key on every attempt. Each attempt looks like a new, distinct request to the external system. The external service — a payment gateway, a shipping provider, an ERP — processes all of them because it has never seen those keys before, resulting in duplicate charges, duplicate shipments, or duplicate records. No exception is raised in Salesforce; all calls return 200 OK.

**When it occurs:** Any outbound callout pattern where the idempotency key is generated at call time rather than at enqueue time. Common in code generated by LLMs or copied from documentation examples that show `UUID.randomUUID()` in the method body rather than as a persisted field value.

**How to avoid:** Generate the idempotency key exactly once, in the code path that creates the work item (the trigger, the flow, or the Queueable enqueue method). Persist it immediately to a custom field on the driving record (e.g., `Order__c.Callout_Idempotency_Key__c`). In the execution path, read the persisted key from the record — never generate a new one. If the field is null (unexpected), treat it as a bug and log an alert rather than generating a fallback key.

## Gotcha 3: External ID Field Not Marked Unique Allows Duplicate Values

**What happens:** An External ID field created with `externalId=true` but without `unique=true` allows multiple records to hold the same External ID value. The first upsert call inserts successfully. A later upsert against the same External ID value encounters two (or more) records with that value and returns `HTTP 300 MULTIPLE_CHOICES` instead of updating the matching record. The integration error handler may not be designed to handle 300-level responses, causing silent failures or incorrect branching.

**When it occurs:** This happens most often when the External ID field is added through the Setup UI with "Unique" unchecked, or when the field definition in a metadata deployment omits `<unique>true</unique>`. It may not be caught in development because the test dataset only has one record per External ID value. The production dataset accumulates duplicates over time and the 300 error only surfaces when a retry happens to hit a duplicated value.

**How to avoid:** Always include `<unique>true</unique>` in the External ID field metadata definition. Verify the setting in Setup → Object Manager → [Object] → Fields & Relationships → [Field] and confirm "Unique" is checked. If the field already has duplicate values in production, the unique constraint cannot be applied until duplicates are resolved — treat this as a data quality incident requiring a deduplification effort before enabling the constraint.

## Gotcha 4: ReplayId Checkpoint Written Before Processing Silently Skips Failed Events

**What happens:** A subscriber stores the ReplayId checkpoint at the beginning of event processing (before DML or callout) as a "mark as seen" pattern. If the processing fails partway through — due to a governor limit, a DML exception, or a callout timeout — the checkpoint already reflects the failed event's ReplayId. On re-subscription, the subscriber starts from after the failed event, which is never re-processed. From the subscriber's perspective the event was "handled"; from the business perspective the operation never completed.

**When it occurs:** In subscriber triggers or flows that write the checkpoint first to prevent re-processing on retry, inadvertently converting at-least-once delivery into at-most-once delivery. This pattern is common in tutorials that emphasize avoiding duplicate processing but do not account for the failure path.

**How to avoid:** Write the ReplayId checkpoint only after all processing for the event batch has completed and committed. If processing fails, do not advance the checkpoint. Accept that the event will be re-delivered on the next subscription and design the processing logic to be idempotent so re-delivery is safe. The External ID upsert pattern in the subscriber body provides the re-delivery safety; the checkpoint provides the resumption position.

## Gotcha 5: Bulk API v2 Upsert Uses Case-Sensitive External ID Matching

**What happens:** The single-record REST upsert endpoint performs case-insensitive matching on Text External ID values. The Bulk API v2 upsert performs case-sensitive matching. An external system that sends `"ORD-001"` while Salesforce stores `"ord-001"` (or vice versa) will find no match under Bulk API v2 and insert a new record instead of updating the existing one, creating a duplicate.

**When it occurs:** When the integration switches from REST single-record upsert (which worked correctly) to Bulk API v2 upsert for performance, without normalizing case. Also occurs when different external systems send the same External ID value with different casing conventions (e.g., one system sends uppercase, another sends lowercase).

**How to avoid:** Normalize External ID values to a canonical casing (uppercase is conventional) at the point of ingestion — in the ETL layer, the middleware, or in a before-insert/before-update trigger on the Salesforce side. Document the canonical casing convention in the field's description and enforce it in the integration contract with the external system.
