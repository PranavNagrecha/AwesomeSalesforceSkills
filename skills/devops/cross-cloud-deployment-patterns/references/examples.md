# Examples — Cross-Cloud Deployment Patterns

## Example 1: Full Multi-Cloud Launch — Sales Cloud + Service Cloud + Experience Cloud Community Portal

**Context:** An enterprise is launching a B2B portal on Experience Cloud alongside a Sales Cloud CPQ rollout and a Service Cloud case management implementation. The deployment contains 200+ components spanning all three clouds, including new custom objects, Apex service classes, permission sets, a Network record, and a DigitalExperienceBundle for the portal site.

**Problem:** The team attempted to deploy everything in a single package. The deployment failed with `Entity of type 'Network' named 'Partner_Portal' cannot be found` during the ExperienceBundle component evaluation. The Network component was in the same package and had been successfully processed by the Metadata API — but ExperienceBundle evaluation happened before the Network record was fully committed, causing the cross-reference to fail.

**Solution:**

Split the release into three ordered deployments:

```xml
<!-- foundation-package.xml — deploy first -->
<types>
  <members>*</members>
  <name>CustomObject</name>
</types>
<types>
  <members>*</members>
  <name>ApexClass</name>
</types>
<types>
  <members>*</members>
  <name>PermissionSet</name>
</types>
```

```xml
<!-- network-package.xml — deploy second, after foundation succeeds -->
<types>
  <members>Partner_Portal</members>
  <name>Network</name>
</types>
<types>
  <members>Partner_Portal</members>
  <name>CustomSite</name>
</types>
```

```xml
<!-- experience-package.xml — deploy third, after network succeeds -->
<!-- SiteDotCom deliberately excluded -->
<types>
  <members>Partner_Portal</members>
  <name>DigitalExperienceBundle</name>
</types>
```

Deploy commands in sequence:

```bash
sf project deploy start --manifest foundation-package.xml --target-org prod --wait 60
sf project deploy start --manifest network-package.xml --target-org prod --wait 30
sf project deploy start --manifest experience-package.xml --target-org prod --wait 30
```

**Why it works:** By the time the DigitalExperienceBundle transaction starts, the Network record `Partner_Portal` already exists and is fully committed in the target org. The cross-reference resolves correctly because the dependency is satisfied by a prior committed deployment rather than by in-flight ordering within a single transaction.

## Example 2: Experience Cloud Site Content Update — SiteDotCom Exclusion

**Context:** A developer retrieved an Experience Cloud site (`Customer_Community`) using `sf project retrieve start` to pull down recent content changes. During the retrieve, Salesforce included `SiteDotCom` metadata alongside the `ExperienceBundle` in the local project. The developer then staged all retrieved files and attempted to deploy to a staging sandbox.

**Problem:** The deployment failed immediately with a generic metadata error. No specific component was cited in the error. The developer assumed the ExperienceBundle XML was malformed and spent several hours inspecting it. The actual cause was the presence of `SiteDotCom` in the deployment manifest, which is not a deployable type in a standard Metadata API deployment package.

**Solution:**

Before building the deploy manifest, audit retrieved metadata for SiteDotCom:

```bash
# Check if SiteDotCom files exist in the retrieved metadata
find ./force-app -name "*.site" -o -name "SiteDotCom" | head -20
```

Exclude SiteDotCom from `package.xml`:

```xml
<!-- Do NOT include this in package.xml -->
<!-- <types><members>Customer_Community</members><name>SiteDotCom</name></types> -->

<!-- Include only the deployable Experience type -->
<types>
  <members>Customer_Community</members>
  <name>ExperienceBundle</name>
</types>
```

Add to `.forceignore` to prevent accidental staging:

```
# Exclude non-deployable SiteDotCom blobs
**/siteDotCom/**
*.siteDotCom
```

**Why it works:** SiteDotCom is auto-generated by the platform and functions as a binary representation of site content. It is not part of the Metadata API's deployable surface. Excluding it from the manifest allows the ExperienceBundle deployment to proceed normally. The SiteDotCom blob is regenerated by the platform after the ExperienceBundle lands.

## Anti-Pattern: Deploying Experience Layer Before Foundation in a Single Transaction

**What practitioners do:** Bundle all cross-cloud metadata — custom objects, Apex, Network, and ExperienceBundle — into a single `package.xml` and deploy in one transaction. This is the natural approach when treating the entire release as one unit and wanting to minimize deployment operations.

**What goes wrong:** The Metadata API does not guarantee that Network is fully committed before ExperienceBundle is evaluated within a single deployment transaction. When ExperienceBundle evaluation happens before the Network record is committed, the deployment throws `Entity of type 'Network' named 'X' cannot be found`. This error looks like a missing-component problem but is actually an ordering race condition within a single transaction.

**Correct approach:** Always split cross-cloud deployments into a minimum of two ordered transactions: (1) foundation and network, then (2) experience layer. This eliminates the intra-transaction race condition because the network record is guaranteed to be committed before the experience deployment transaction begins.
