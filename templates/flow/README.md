# Flow shared templates

| File | When to use |
|---|---|
| `RecordTriggered_Skeleton.flow-meta.xml` | Starting a new record-triggered flow. Has entry-criteria decision + before-save trigger |
| `FaultPath_Template.md` | Reference for adding fault paths to *any* Create/Update/Delete/Action element — declarative runbook |
| `Subflow_Pattern.md` | Anytime a flow calls another flow — defines input/output contract rules |

## Why these live as `.md` more than `.flow-meta.xml`

Flow metadata is verbose, auto-generated by Flow Builder, and rarely hand-
edited productively. The `RecordTriggered_Skeleton.flow-meta.xml` is the
minimal valid scaffold; everything else is captured as prose + pseudo-XML
because that's the form teams actually read.

## Fault path rule

Every Create, Update, Delete, and Action element **must** have a fault path.
A flow without fault paths is a silent-failure flow. See `FaultPath_Template.md`
for the canonical log-and-notify pattern — it writes to the same
`Application_Log__c` object that Apex `ApplicationLogger` uses, giving ops
one queryable surface for declarative + programmatic errors.
