# MASTER_QUEUE.md — Universal Salesforce Skill Build Queue

> **The row data formerly in this file moved to [`BACKLOG.yaml`](BACKLOG.yaml)
> on 2026-05-01.** The full git history of the old tables is still available
> via `git log --diff-filter=D` if you ever need to recover a row.

## Purpose

Single source of truth for the full Role × Domain × Cloud skill matrix.
This file drives the expanded vision: every Salesforce role, every cloud,
every task.

## Vision

One public repo. Every Salesforce professional (Admin, BA, Dev, Data,
Architect) drops it into Claude, Cursor, or any AI and gets
role-appropriate, task-accurate Salesforce guidance grounded in official
docs — not training-data guesses.

## Where the data lives now

| File | What it holds | Edited by |
|---|---|---|
| [`BACKLOG.yaml`](BACKLOG.yaml) | Machine-readable queue — every pending / in-flight / duplicate / blocked entry. | `scripts/queue_reader.py` (atomic updates) or careful hand-edit. |
| [`docs/queue-progress.md`](docs/queue-progress.md) | Generated dashboard — status counts, drift, oldest TODO, next-pick. Read-only. | `scripts/generate_queue_dashboard.py` (regenerated by `skill_sync.py`). |
| `skills/<domain>/<name>/` | Authoritative truth for what's already built. | `scripts/new_skill.py` then fill the scaffold. |

DONE state is **not** tracked in `BACKLOG.yaml` — the filesystem under
`skills/` is authoritative. The dashboard reconciles the two and surfaces
drift (e.g. queue entries still TODO when the skill is already on disk).

## Agent workflow (read before doing anything)

1. Read `CLAUDE.md` and `AGENT_RULES.md` first. This file does not override them.
2. Pick an entry:
   ```bash
   python3 scripts/queue_reader.py --next --status RESEARCHED,TODO
   ```
   Prefer RESEARCHED (notes already captured) over plain TODO.
3. Claim it:
   ```bash
   python3 scripts/queue_reader.py --set-status IN_PROGRESS \
     --id <entry-id> --actor "<agent>@<host>"
   ```
4. Confirm coverage doesn't already exist:
   ```bash
   python3 scripts/search_knowledge.py "<skill-name>" --domain <domain>
   python3 scripts/audit_duplicates.py --domain <domain>      # full pairwise
   ```
   If covered, set status DUPLICATE with a pointer to the existing skill.
5. Read the official sources for this domain from
   `standards/official-salesforce-sources.md` BEFORE writing anything.
6. Scaffold:
   ```bash
   python3 scripts/new_skill.py <domain> <skill-name> --strict
   ```
   `--strict` blocks scaffolding when the proposed name produces a
   near-duplicate of an existing skill (recommended for agents).
7. Fill every `TODO:` marker in every generated file with real,
   doc-grounded content. SKILL.md body must be 300+ words. Description
   must include `NOT for ...`. Triggers need 3+ natural-language symptom
   phrases, each 10+ chars.
8. Sync:
   ```bash
   python3 scripts/skill_sync.py --skill skills/<domain>/<skill-name>
   ```
   Fix every ERROR. Do not use `--skip-validation`.
9. Add a query fixture to `vector_index/query-fixtures.json`. Run
   `python3 scripts/search_knowledge.py "<query>"` and confirm the new
   skill appears in the top 3.
10. ```bash
    python3 scripts/validate_repo.py
    ```
    Must exit 0.
11. Don't mark the entry DONE — drop it from `BACKLOG.yaml` instead. The
    filesystem is authoritative for built skills.
12. Commit: skill package + `registry/` + `vector_index/` +
    `docs/SKILLS.md` + the BACKLOG.yaml deletion + the regenerated
    `docs/queue-progress.md`.

## Status key

| Status | Meaning |
|---|---|
| `TODO` | Not started; not yet researched. |
| `RESEARCHED` | Deep research done, notes captured. Ready to build next session. |
| `RESEARCH` | Investigation needed before deciding TODO vs DUPLICATE. |
| `IN_PROGRESS` | Claimed by an agent — do not touch. |
| `DUPLICATE` | Skill already exists. Kept for traceability. |
| `BLOCKED` | Stopped. Read entry notes for the blocker. |
| `UPDATE` | Existing skill needs refresh. |
| `SHIPPABLE` | Ready for release / promotion. |

`DONE` is allowed for transitional updates but the dashboard ignores it
(filesystem is authoritative).

## Updating the queue programmatically

Every operation goes through `scripts/queue_reader.py`. The CLI is the
contract; the YAML format may evolve, but these commands stay stable:

```bash
python3 scripts/queue_reader.py --summary
python3 scripts/queue_reader.py --list
python3 scripts/queue_reader.py --next --status TODO,RESEARCHED
python3 scripts/queue_reader.py --set-status DONE \
  --id <entry-id> --actor "<agent>@<host>"
```

To recover the historical 1500-row table layout (e.g. for archaeology):

```bash
git show HEAD:MASTER_QUEUE.md     # this file before 2026-05-01
git log --follow -p MASTER_QUEUE.md
```

The historical row data was migrated to `BACKLOG.yaml` via
`scripts/_migrations/migrate_queue_to_yaml.py` — re-running it rebuilds
BACKLOG.yaml deterministically from a pre-migration MASTER_QUEUE.md.
